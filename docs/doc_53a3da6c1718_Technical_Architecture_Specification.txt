================================================================================
PROJECT: INFINITE NARRATIVE (LLM Text Adventure)
================================================================================

1. OVERALL TECH STACK
================================================================================

Core Language & Runtime:
- Go 1.21+ (Primary backend language)
- Standard library (net/http, context, database/sql, encoding/json)
- Chi Router (Lightweight HTTP routing for API endpoints)

Database & Storage:
- PostgreSQL 15+ (Primary persistence for production)
- SQLite (Embedded alternative for edge/low-resource deployments)
- SQLx (Helper library for strict SQL scanning, avoiding ORM magic)

LLM Integration:
- OpenAI API (GPT-4o or GPT-4-Turbo for best reasoning)
- Go client for OpenAI API (or custom http.Client implementation)

Infrastructure & Deployment:
- Docker (Containerization)
- Docker Compose (Orchestration)
- Alpine Linux (Base image for minimal size)
- Multi-arch builds (amd64, arm64) for Raspberry Pi/NAS compatibility

Frontend:
- Vanilla JavaScript + HTML5 (Served via Go embedded filesystem)
- Server-Side Rendering (Minimal) + Client-side fetch for game loop

Security & Validation:
- Input validation middleware (Strict regex for choices)
- Environment variable management (Godotenv for local dev)
- Structured logging (Zap or slog)

================================================================================
2. DATABASE DESIGN
================================================================================

Strategy: Relational schema to ensure data integrity and easy reconstruction of the story history.

Table: users
- id (UUID, PK)
- username (VARCHAR, Unique)
- created_at (TIMESTAMP)
- last_active (TIMESTAMP)

Table: sessions
- id (UUID, PK)
- user_id (UUID, FK)
- genre (VARCHAR) -- e.g., "adventure", "romance", "cyberpunk"
- player_name (VARCHAR)
- player_age (INTEGER)
- player_gender (VARCHAR)
- current_state (JSONB) -- Flexible storage for HP, Inventory, Stats
- status (VARCHAR) -- 'active', 'finished', 'abandoned'
- created_at (TIMESTAMP)
- updated_at (TIMESTAMP)

Table: story_turns
- id (UUID, PK)
- session_id (UUID, FK)
- turn_number (INTEGER)
- narrative_text (TEXT) -- The story segment generated by LLM
- choices (JSONB) -- Array of options: [{"key": "A", "text": "..."}, ...]
- selected_choice (CHAR(1)) -- The user's input
- llm_reasoning (TEXT, Optional) -- Why the state changed (for debugging)
- created_at (TIMESTAMP)

Indexes:
- INDEX on sessions(user_id)
- INDEX on story_turns(session_id, turn_number)

JSONB Structure for current_state (Example):
{
  "health": 100,
  "max_health": 100,
  "inventory": ["Rusty Sword", "Health Potion"],
  "stats": {
    "strength": 10,
    "charisma": 5
  },
  "location": "Tavern",
  "currency": 50
}

4. ARCHITECTURAL DECISIONS
================================================================================

Decision 1: Structured LLM Output (Deterministic State Management)
Context: The LLM needs to update inventory/health autonomously based on the story flow.
Rationale: To ensure the backend remains the source of truth and prevents "hallucinated" game states from breaking the app, we will enforce a Structured Output (JSON mode) from the OpenAI API.
Implementation:
- The System Prompt will instruct the LLM to return a specific JSON schema containing:
  1. narrative_text (string)
  2. choices (array of 4 strings)
  3. state_update (object with specific keys like health_delta, inventory_add, inventory_remove)
- The Go backend parses this JSON. If parsing fails or the schema is invalid, the backend rejects the turn and asks the LLM to retry (with a tighter constraint prompt) before notifying the user.
- This balances "Autonomous LLM management" with "Backend Correctness."

Decision 2: Modular Monolith with Worker Pools
Context: Support 10+ simultaneous users on limited hardware (Edge/RPi).
Rationale: Microservices add too much operational overhead for a single binary deployment. A monolith is simpler, but we must prevent blocking I/O (LLM API calls) from stalling the HTTP server.
Implementation:
- The HTTP layer accepts requests and pushes jobs to a buffered Channel.
- A pool of worker Go routines (configurable, e.g., 5-10 workers) reads from the channel and processes the LLM interactions.
- This provides natural backpressure and rate limiting, ensuring the API keys aren't spammed and the server remains responsive under load.

Decision 3: Database-First State Management
Context: "Never ending" story requires persistence.
Rationale: Storing state only in memory risks data loss on restart. Since the text adventure is turn-based, latency is not critical (users can wait 1-2 seconds for the AI).
Implementation:
- Every turn is a transaction.
- 1. Fetch current session state.
- 2. Send state + user choice to LLM.
- 3. Receive LLM response.
- 4. Update session state (JSONB merge).
- 5. Insert new story_turn.
- 6. Commit.
- If the LLM crashes or times out, the transaction rolls back, and the user can retry the turn.

Decision 4: Strict Server-Side Validation
Context: User input must be restricted to A, B, C, or D.
Rationale: Client-side validation can be bypassed. Sending arbitrary text to the LLM increases cost and risk of prompt injection/jailbreaks.
Implementation:
- Middleware on the POST /turn endpoint validates the payload.
- Regex: `^(?i)[a-d]$`.
- Any other input returns 400 Bad Request immediately before hitting the LLM service.

Decision 5: Genre-Based Prompt Engineering
Context: Mechanics must adapt to genre (Romance vs Adventure).
Rationale: A generic prompt produces generic stories.
Implementation:
- The `domain/prompts.go` package will hold a map of System Prompts keyed by genre.
- Example: "Adventure" prompt emphasizes HP, loot, and combat. "Romance" prompt emphasizes relationship meters and dialogue.
- The `game_master` service selects the correct prompt template based on the session's genre field.

Decision 6: Text File Generation via Stream
Context: User wants to download the full story as TXT.
Rationale: Loading the entire story into RAM could be memory-intensive if a user plays for 10,000 turns.
Implementation:
- The `story_compiler` service opens a write stream (http.ResponseWriter).
- It queries the `story_turns` table in batches (e.g., 100 turns at a time) ordered by turn_number.
- It writes directly to the stream and flushes, keeping memory usage constant (O(1)) regardless of story length.

Decision 7: Deployment Strategy
Context: Self-hosted, Edge devices.
Rationale: Kubernetes is too heavy for a Raspberry Pi or home NAS.
Implementation:
- Single Docker container running the Go binary.
- Database depends on environment:
  - Production: Connect to external Postgres instance.
  - Edge/Dev: Use an embedded SQLite file (specified in .env).
- Configuration via Environment Variables only (no config files).
